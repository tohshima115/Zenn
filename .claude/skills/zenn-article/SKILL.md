# Zenn記事を書く

ユーザーが話した内容・メモ・アイデアをもとに、この Zenn リポジトリ用の記事（Markdown）を作成します。

## 出力先と形式

**ファイルの置き場所：**
```
articles/<slug>.md
```

**slug（ファイル名）のルール：**
- Zenn の制約：半角英数字・ハイフン・アンダースコアのみ、12〜50文字
- 迷ったら `npx zenn new:article` でランダムなslug（例：`bec77cad7ff3d4.md`）を生成してから中身を書き換える
- 内容がひと目でわかる英語スラグ（例：`why-i-tried-claude-code-for-a-week`）でもよい。どちらでも可

**frontmatterテンプレート：**
```yaml
---
title: "タイトル"
emoji: "😊"
type: "tech" # tech: 技術記事 / idea: アイデア・エッセイ的な内容
topics: ["タグ1", "タグ2"]
published: false
---
```

**各項目の決め方：**
- `emoji`：記事本文を書く前に、5つの候補をユーザーに提示して選んでもらう。内訳は「本命2つ（記事のテーマを素直に表す）」「対抗2つ（別の角度・別のニュアンスで表す）」「突然変異1つ（内容と関係ないくらいふざけた/意外性のあるもの）」。ユーザーが選んだものを frontmatter に反映する。特に指定がなければ本命の1つ目を採用してよい
- `type`：手順・実装・検証など技術的な内容が中心なら `tech`。気づき・考え方・エッセイ寄りの内容なら `idea`。判断に迷う場合は `tech` をデフォルトにしつつ、内容次第で `idea` を提案する
- `topics`：固定のタグリストは使わない。記事の内容に応じてそのつど選ぶ。Zenn の制約で半角英数字のみ・最大5個（ハイフンやスペース、日本語は不可）。よく使う候補：`claude-code` `ai` `vibecoding` `typescript` `nextjs` `cloudflare` など、内容に合わせて選定する
- `published`：常に `false` で作成する。中身をユーザーが確認してから、ユーザー自身が `true` に変更するか、`npx zenn preview` で確認してから公開する

## 本文フォーマットのルール

- セクションの区切りには `##` 見出しを使う（`---` 水平線は使わない）
- 見出しは体言止めで短くつける（例：「本を読んだのに、気づけなかった」）
- コードやコマンドを示す際はコードブロックに言語を指定する（例：` ```bash `）
- Zenn独自の記法が使える場面では活用する（一般的なMarkdownエディタ向けの記法と混同しないこと）：
  - メッセージ：`:::message` / `:::message alert`
  - 折りたたみ：`:::details タイトル`
  - ファイル名付きコードブロック：` ```ts:filename.ts `
  - diffハイライト：```` ```diff ```` のコードブロック内で、行頭に `+` を追加行、`-` を削除行としてつける（例：`+  console.log("added")`）。VitePress/Shikiの `// [!code ++]` 形式は Zenn では効かないので使わない
  - ただし無理に使う必要はない。文章の邪魔にならない範囲で

## 文体のルール

**です・ます調で書く。**
である調は堅苦しいので使わない。ただし「まあ、そういう話でした」みたいな結びや、引用・感嘆のような1文に限っては体言止めや断言もOK。

---

## 書き方の黄金律（高評価される失敗談・体験談の法則）

Qiita・note・Zennで共感・いいねを集める記事には、共通するパターンがある。以下を参考にして記事を構成すること。

### 1. タイトルで「自分ごと」にさせる

- 「失敗」「やらかし」「気づいた」「正直に言うと」など脆弱性ワードを入れる
- 時間軸を明示すると読者が引き込まれる（「〇〇しながら気づいたこと」「今回〜してみた話」）
- 上から目線の「〜すべき」「〜の正解」はNG。「〜と思った話」「〜だった」で終わる形が良い

### 2. 冒頭3行で「あるある感」を出す

- 「あー、これ自分もやったな」と思わせる具体的な失敗・状況を書く
- 抽象的な話から始めない。「このリポジトリのCIが最初は全部落ちてました」みたいな具体的な一言から入る
- 「なぜこの記事を書くか」を説明しない。いきなり本題から入る

### 3. 感情を「説明」せず「再現」する

- NG：「不安でした」「大変でした」
- OK：「あ、これ絶対違うわ、と思いながらリロードを繰り返してました」
- 心の声・独り言・当時の判断ミスを素直に書く
- 「結果としてうまくいった」より「これ違うなと思った瞬間」の方が共感される

### 4. 失敗を隠さない、でも自虐になりすぎない

- 「こんなことをやってしまった」という事実は書く
- ただし過剰な反省や謝罪モードにならない。「まあそれはそれでよかったんですけどね」くらいの距離感
- 失敗は「こうしたら改善した」に繋げる。ただし「学びまとめ」で終わらなくていい

### 5. 上から目線を徹底的に排除する

- 「〜すべき」「〜が重要です」「〜を意識しましょう」は使わない
- 「私の場合は」「自分が今回感じたのは」と限定して書く
- 読者に教えようとしない。「こういうことがあった」を共有するスタンス
- 技術記事（type: tech）で手順を説明する場合も、断定口調より「自分はこうした」という書き方に寄せる

### 6. 結論・まとめを作りすぎない

- 「まとめ」セクションを無理に作らない
- 最後の段落が自然に締めになっていればOK
- 「そんな感じの話でした」「まあ、そういうことを思いながら作っていました」みたいな余韻で終われる
- 「以上です」「参考になれば幸いです」は使わない

### 7. 構成は「事実 → 感じたこと → なぜそう思ったか → 今どう考えているか」

流れの例：
1. こういうことがあった（具体的な出来事）
2. そのとき自分はこう感じた（感情の再現）
3. なぜそうなったか・なぜそう思ったか（自分なりの分析）
4. 今はこう思っている（現時点の考え、あるいは「まだわからない」でもOK）

---

## 記事作成の手順

1. ユーザーの話した内容・メモからキーとなる体験・気づきを抽出する
2. `type`（tech / idea）を判断する
3. タイトルを考える（上記タイトルの黄金律を参照）
4. emoji候補を5つ（本命2・対抗2・突然変異1）提示してユーザーに選んでもらう
5. 本文を書く（です・ます調、黄金律に沿って。Zenn独自記法を活かせる場所があれば使う）
6. frontmatter を埋める（選ばれた `emoji`、`type`、`topics` を決定、`published` は常に `false`）
7. slug を決める（`npx zenn new:article` で生成するか、内容が分かる英語スラグを自分で決める）
8. `articles/<slug>.md` に保存する
9. 内容をユーザーに確認してもらう。コミットは明示的に指示された場合のみ行う（メッセージ例：`feat: 〇〇の記事を追加`）
