ni、いいですね。

最近重い腰を上げて @antfu/ni を使うようになって、地味に効くな～と思っています！

## 前提

自分は現在 Web 制作会社で働いているフロントエンドエンジニア（？）です。最近は AI のおかげで、コーディングがメイン業務じゃない人がコードを微修正する機会も増えてきました。そんなことでデザイナーに技術的な支援もしたりしてます。

うちの会社は外部のコーダーに基本的な実装をお願いする場面も多く、フレームワークは Astro で書いてもらうようにお願いしています。Astro は React・Vue（なんならVanillaJS）など好きな UI ライブラリを選べる作りになっていて、ディレクトリ構成自体はある程度型が決まるので選びました。UI ライブラリの習熟度の差をある程度吸収しつつ、モダンな開発に慣れている人もレガシーに慣れている人も、一定の統一感を持って制作できる体制にはしています。

## パッケージマネージャーだけ、なんかブレる

ただ、パッケージマネージャーだけはなんか、ブレがあるんです。自分は pnpm を推奨していますが、npm でやっている人もいますし、今後 bun がいいと言う人が出てくるかもしれません。

ここを「pnpmで統一してください」と指示するのは、なんか違うなとずっと思ってます。強制するほどのことでもないというか、それぞれに慣れたやり方があるので。

でもそうなると、

- `pnpm i`をすると、pnpm と npm の lock ファイルが両方できてしまったりする
- 非エンジニアに「lock ファイルを見てコマンドを変えてください」と教えるのは負荷が高い

という状態になります。

lock ファイルが複数残るのはなんとなく気持ち悪いだけじゃなくて、Cloudflare へのデプロイ時にたまにまずいことになります。Cloudflare のようなホスティング先は、プロジェクト内の lock ファイルを見てどのパッケージマネージャーを使うか自動で判定するので、複数残っていると意図していない方が選ばれてビルドが落ちる、ということもあるみたいです。

## ni を試してみた

そういえば、こんなツールがあったなと思い出したのが `@antfu/ni` でした。lock ファイルを自動で検出して、対応するパッケージマネージャーのコマンドを実行してくれるツールです。前から知ってはいたんですが、これまではpnpmしか使わないし、問題が起こったら対応できるしいいかで導入しようとまではしていませんでした。今回非エンジニアの開発環境を整える機会があって、もしかしたらここで躓く可能性があるかと思ったのと、たしかそれを解決するniとかいうツールあったよなと思い出し試してみたら、実際良かったです。

```bash
npm i -g @antfu/ni
```

```bash
ni          # install（npm / yarn / pnpm / bun を自動判別）
ni vite     # add
nr dev      # run dev
nun vite    # uninstall
```

対応するパッケージマネージャーが入っていなければ、ni が自動でインストールしてくれるので、bun を使ったプロジェクトが来ても困りません。「`ni` と `nr dev` だけ覚えてください」と言えるようになったのは、地味に楽です。

## PM を統一させるんじゃなくて、こっち側で吸収する

ni 自体は前から頭の片隅にあったツールですが、AI のおかげで非エンジニアやデザイナーが開発に関わる機会が増えた今だからこそ、刺さる場面が増えている気がします。

パッケージマネージャーを統一させるのではなく、パッケージマネージャーのブレを環境構築の時点で吸収する。そういう考え方のツールが、今っぽく感じました。
