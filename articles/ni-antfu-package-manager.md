## ni とは

`@antfu/ni` は、プロジェクトの lock ファイルを見て、npm / yarn / pnpm / bun のうち「正しいコマンド」を自動で選んで実行してくれるツールです。

```bash
npm i -g @antfu/ni
```

```bash
ni          # install（lock ファイルから自動判別）
ni vite     # add
nr dev      # run dev
nun vite    # uninstall
```

`pnpm i` なのか `npm install` なのかを気にせず、`ni` と打てば入る。`nr dev` で dev サーバーが立ち上がる。覚えることはこれくらいです。対応するパッケージマネージャーが入っていなければ ni が入れてくれるので、bun のプロジェクトが来ても困りません。

これだけのツールなんですが、自分の状況にはこれがかなり刺さりました。

## 前提：フレームワークは決められても、PM はブレる

自分は Web 制作会社でフロントエンドの技術支援をしている立場です。外部のコーダーやデザイナーもコードを触るので、フレームワークは Astro に寄せて、ある程度みんなが同じ型で書けるようにしています。

ただ、パッケージマネージャーだけはどうしてもブレるんです。自分は pnpm 推奨ですが、npm の人もいるし、今後 bun がいいと言う人も出てくるかもしれません。ここを「pnpm で統一してください」と強制するのも違うなと思っていて、それぞれ慣れたやり方があるので。

### ブレると何が地味に困るか

- `pnpm i` したつもりが npm でやられて、lock ファイルが両方できたりする
- lock ファイルが複数残ると、Cloudflare のようなホスティング先が「どのパッケージマネージャーか」を lock ファイルから自動判定するので、意図しない方が選ばれてビルドが落ちることがある
- 何より、非エンジニアに「lock ファイルを見てコマンドを変えてください」と教えるのは、本質的じゃない学習コストが無駄に乗る

最近は AI のおかげで、コーディングがメイン業務じゃない人がコードを微修正する機会も増えました。そういう人に環境構築でつまずいてほしくない、というのが一番大きいです。

## PM を統一するんじゃなく、環境側で吸収する

ni を入れておけば、「`ni` と `nr dev` だけ覚えてください」で済みます。パッケージマネージャーを揃えさせるのではなく、ブレを環境構築の時点で吸収してしまう。この発想が、非エンジニアも開発に関わる今の状況にはちょうどよかったです。

これは非エンジニアに限らず、自分自身の認知負荷も地味に下がります。プロジェクトを開くたびに「これ `package-lock.json` だっけ、いや `pnpm-lock.yaml` か…」と確認して頭を切り替える、あの小さな引っかかりが要らなくなる。どのプロジェクトでも `ni` と打てば入る、と考えなくていいのは普通に楽です。

## おまけ：Windows（PowerShell）での初期設定

ひとつ注意点があって、PowerShell では `ni` が `New-Item` のエイリアスとして最初から予約されています。そのまま `ni` と打つと ni ではなく New-Item が動いてしまうので、エイリアスを外す必要があります。

一時的に外すなら：

```powershell
Remove-Item Alias:ni -Force
```

毎回効かせたいなら、PowerShell のプロファイルに書いておきます。

```powershell
notepad $PROFILE
```

開いたファイルに以下を追記して保存すれば、次回以降のセッションから `ni` が使えます。

```powershell
Remove-Item Alias:ni -Force -ErrorAction SilentlyContinue
```

macOS / Linux ではこの手順は不要です。Windows の人だけ、ここだけ気をつけてください。
